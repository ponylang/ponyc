# Stress-test concurrency: two lanes cap the scheduled loop at 2 at a time

A GitHub concurrency group admits only one run, so "no more than two stress workflows at once" is enforced by *two* repo-wide groups, not by a count or by cron spacing. The three tcp-swarm workflows (`stress-test-tcp-swarm-{linux,macos,windows}.yml`) share **`stress-scheduled`**; the six generative workflows (`stress-test-generative-{systematic,normal}-{linux,macos,windows}.yml`) share **`stress-scheduled-generative`**. One run per group ⇒ at most two concurrent. A manual `workflow_dispatch` run instead gets a unique `…-${{ github.run_id }}` group so a branch check never queues behind the scheduled loop — this relies on every stress workflow having exactly the two triggers (`schedule` + `workflow_dispatch`); the per-event `(failure() || cancelled()) && github.event_name == 'schedule'` Zulip alert and the `--skip-on-miss` libs path assume the same split. **A new stress workflow MUST join one of the two groups** (or the 2-at-a-time cap silently breaks); the cron slots are merely spread across the day (the website's `scheduled-jobs.md` tracks them — keep it in sync) and do not enforce the cap. With `cancel-in-progress: false` on every workflow, GitHub keeps only one run pending per group and discards older pending ones, so if two crons in the same lane ever collided a scheduled run would be silently *dropped* (and the Zulip alert never fires for a run that never ran — `cancelled()` needs a job that started and was then cancelled, not one dropped before it began) — keep each lane's slots well spaced.

**The tcp-swarm lane** runs `test/rt-stress/tcp-swarm/orchestrate_tcp.py` (not a `make test-stress-*` target — the retired tcp-open-close trio used make targets, since removed). Linux and macOS fire **twice a day** and windows **once** (its msys2 install + build make a run long, so it takes the single wide afternoon gap), and the tcp-swarm crons are placed in the **gaps between the generative crons** to keep the two lanes from overlapping on the wall clock in the common case: generative runs at 02/08/11/14/18/22 UTC, tcp-swarm in the clear windows between them (06/12 linux, 09/15 macos, 16:30 windows). This is best-effort, not a guarantee — a generative job can run up to its 240-min (4h) cap, longer than the 3h spacing to the next generative cron, so a long generative run *can* still be in flight when a tcp-swarm cron fires. The hard guarantee is only what the two groups give: at most one run per lane, so at most two stress jobs at once (and they land on separate GitHub-hosted runners, so an overlap doesn't contend for a machine). Each seed runs under **lldb** so a crash yields a backtrace (a normal, non-reproducible run's in-the-moment stack is the only artifact), which is why the linux container adds `--cap-add=SYS_PTRACE --security-opt seccomp=unconfined` and the windows job keeps the msys2/lldb `Install Dependencies` step. The tcp-swarm jobs cap at `timeout-minutes: 200` (headroom over a ~50-min `--budget-seconds` window of starting seeds, each run to completion, plus a 90-min non-failing `--timeout-seconds` backstop for a final over-long seed, plus build -- the ceiling must exceed budget + backstop + build so a healthy backstop-bound final seed can't trip the job timeout, which would be a false failure). At 200 min a job *can* outlast the tightest tcp-swarm cron gap (15:00 macOS → 16:30 windows is 90 min) and still be running when the next tcp-swarm cron fires — the shared `stress-scheduled` group then queues that run, and in the worst case (three firings inside one job's length) drops the older pending one. That is deliberately accepted: an overlap, or a dropped scheduled run, is preferred over cutting a healthy run short and reporting it as a failure — the run mechanism in `orchestrate_tcp.py` fails a seed only on a no-progress hang, never for running long. The 2-at-a-time cap the two groups give is unchanged. They upload their failure bundles (`tcp-swarm-bundles-*`) as an artifact.

**The generative lane** runs in two modes, each its own orchestrator under `test/rt-stress/generative/`, and both upload their failure bundles: **systematic** builds a systematic ponyc — `use=scheduler_scaling_pthreads,systematic_testing` on linux-glibc/macos-arm, `use=systematic_testing` alone on windows (which scales the scheduler with native primitives, not pthreads; output dir `build\debug-systematic_testing`) — and runs `orchestrate_systematic.py` (serialized, reproducible; linux-glibc + macos-arm + windows); **normal** builds a plain `config=debug` ponyc (no use flags) and runs `orchestrate_normal.py` (the real, multi-threaded runtime; linux-glibc + macos-arm + windows). Both modes run each seed under **lldb** on the POSIX lanes (same SYS_PTRACE reason as tcp-swarm; a crash or watchdog timeout then carries a backtrace in the bundle — see `.known-couplings/lldb-timeout-abort-capture.md`); the windows systematic job passes `--no-lldb` and runs direct (its lldb capture is unverified and deferred), while the windows normal job keeps its msys2/lldb install. Each generative job caps at `timeout-minutes: 240`, under twice the tightest 3h generative cron gap, so a job that runs to its cap is cancelled before it can span two firings and drop a run; at worst it delays the next.

All the libs-cache platforms (ubuntu26.04-x86, arm64-macos, x86-windows) are warmer Stage-1 PR platforms, so the stress jobs need no warmer change of their own.

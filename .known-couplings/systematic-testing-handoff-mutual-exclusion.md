# The systematic-testing handoff must run under the lock on every platform, waker included

Under `use=systematic_testing` the scheduler serializes execution to one thread at a time. That single-runner invariant is not free: the handoff in `src/libponyrt/sched/systematic_testing.c` — set the shared `active_thread` to the next thread, wake it, then park the current thread — must be mutually exclusive, or the waker and the thread it just woke both run for a window. The running thread enforces this by holding a lock continuously the entire time it is "running" (acquired in `ponyint_systematic_testing_wait_start` / `ponyint_systematic_testing_start`, released only inside the condition-variable wait while parked, and finally released when the thread stops). On the `scheduler_scaling_pthreads` path the lock is `systematic_testing_mut` (mutex + per-thread `pthread_cond`); on the Windows path it is `systematic_testing_win_lock` (`SRWLOCK` + a single shared `CONDITION_VARIABLE`, woken with `WakeAllConditionVariable` since the wake targets no specific thread and every waiter re-checks `active_thread == self`).

Two things silently reintroduce an intermittent Windows hang if you get them wrong. First, the waker must not touch shared handoff state after waking the next thread unless it still holds the lock — most sharply, the yield cleanup branch frees `threads_to_track`, and reading `current_thread->stopped` (or parking on `active_thread`) after an unlocked wake is a use-after-free once the woken thread races ahead, stops last, and runs cleanup. Second, the wake and the park must stay under the same lock: the woken thread cannot return from its own cond-wait until it reacquires the lock, which the waker holds until it parks — remove or re-scope the lock and the two run concurrently. The Windows build is the one systematic config not built with pthreads (the `static_assert` in `systematic_testing.h` requires pthreads everywhere else), so it is the config that regresses; a hit surfaces only as a watchdog timeout with no thread state.

This is **not** caught per-PR: nothing in normal CI builds `systematic_testing`. The only standing coverage is the generative systematic stress jobs (`.github/workflows/stress-test-generative-systematic-*.yml`, daily) — and they catch it only probabilistically, because the trigger is a wake landing in the concurrent window, which may not occur on any given run. Run (Windows, where the bug lives): `.\make.ps1 -Command configure -Config Debug -Use systematic_testing && .\make.ps1 -Command build -Config Debug`, then loop the generative harness under `test/rt-stress/generative` under parallel load with `--ponymaxthreads` >= 2. Run (pthreads path): `make configure config=debug use=scheduler_scaling_pthreads,systematic_testing && make build config=debug`, then the generative harness and `python3 .ci-scripts/systematic-testing/determinism_smoke.py`.

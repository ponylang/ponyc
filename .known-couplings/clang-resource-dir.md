# Clang resource directory: gencshim's binary-relative offsets must match packaging

`clang_resource_dir` (`src/libponyc/codegen/gencshim.cc`) locates clang's builtin headers at fixed offsets from the ponyc binary: `../lib/clang/<major>` (installed) and `../libs/lib/clang/<major>` (build tree). The installed copy is placed by the root CMakeLists `install(DIRECTORY ...)` rule, which produces the versioned `lib/pony/<version>/` tree on every platform (the binary sits at `.../lib/pony/<version>/bin`, clang at `.../lib/pony/<version>/lib/clang`, so the `../lib/clang` offset holds); the Makefile `install` target still copies the same layout on Linux/macOS until it is retired (#5588). LLVM installs the resource dir under `lib/` or `lib64/` depending on distro, so the rule copies whichever exists. Moving where packaging or the libs build puts `lib/clang` breaks C shim compilation with "unable to find clang's resource directory".

The same binary-relative coupling applies to ponyc's own headers for shims (`add_pony_include_args`). Installed offsets: `../include/pony` for the headers a shim names directly (`pony.h`, `ponyassert.h`, `platform.h`, `threads.h`, `paths.h`), plus `../include` for `pony.h`'s transitive `#include <pony/detail/atomics.h>` which resolves to `include/pony/detail/atomics.h`. Both offsets are required; the order between them doesn't change resolution (no shim header is reachable from both roots). What is load-bearing is that these dirs are pushed AFTER the system include dirs (`add_system_include_args` runs first), so a shim's direct `#include <threads.h>`/`#include <paths.h>` resolves to the system header, not the copy under `pony/`; pushing ponyc's dirs ahead of the system dirs would reintroduce the #5613 shadowing for shims. The root CMakeLists `install(FILES ...)` rules put these under `<version>/include/pony` (and `/pony/detail`) in the versioned tree on every platform; the Makefile `install` target still copies the same on Linux/macOS until it is retired. Build tree: `../../src/libponyrt` (pony.h) + `../../src/common` (ponyassert.h and pony/detail/atomics.h). The headers live under `pony/` so the install dir can double as a shared system include (DESTDIR packaging, Windows) without dropping a flat collision-prone name there — `paths.h` would shadow the system `<paths.h>`, `threads.h` the C11 `<threads.h>` (#5613); they are deliberately NOT installed into the shared `$(prefix)/include`. `pony.h` and `pony_assert` (`<ponyassert.h>`) are supported for shims; `platform.h`/`threads.h`/`paths.h` are ponyassert.h's internal include closure, not a committed interface. Moving or renaming the installed `pony/` layout — including separating `pony.h` from its co-located `pony/detail/atomics.h` — breaks shim compilation.

The gtest suite only exercises the build-tree offsets. Run: the CShimTest suite in `libponyc.tests`; on packaging changes, also verify a shim that does `#include <ponyassert.h>` builds against a `make install` tree.

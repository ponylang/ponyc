# The three ASIO backends must each yield to the systematic-testing scheduler the same way

Under `use=systematic_testing` the ASIO event-loop thread is one of the cooperatively-scheduled threads, so when it gets its turn it must promptly hand control back rather than block on I/O — otherwise, because only one thread runs at a time, the whole interleaving stalls (no actor ever runs, and shutdown coordination — which must wake the ASIO thread so it observes the stop request and exits — never completes). Each backend's wait loop therefore needs two things in step: (1) a **non-blocking poll** — a zero timeout — instead of the production infinite wait (`epoll.c` `wait_time = 0` vs `-1`; `kqueue.c` a zeroed `timespec`, `ts.tv_sec = ts.tv_nsec = 0`, vs a NULL `timeout`; `sock_notify.c` `wait_ms = 0` vs `INFINITE`), and (2) a `SYSTEMATIC_TESTING_YIELD()` on the **reachable** path right after the wait returns. Both are no-ops in non-systematic builds (`SYSTEMATIC_TESTING_YIELD()` expands to nothing and the timeout stays infinite).

The timeout must be **zero**, not merely finite. Because execution is serialized to one thread at a time, a real wait on the ASIO turn stalls the entire program for that long on every ASIO turn, and the ASIO thread takes a large share of the round-robin turns. A small non-zero value (this was 10ms) multiplies into wall-clock time: a workload with many scheduling steps spends nearly all of its wall-clock asleep on ASIO's turns, taking minutes for seconds of work and tripping watchdogs, so it looks like a hang even though it is only slow. Zero polls and hands the turn straight back, so systematic wall-clock reflects the work rather than the wait. Nothing is lost by not waiting: `epoll_wait`/`kevent`/`GetQueuedCompletionStatusEx` with a zero timeout still report events that are already ready, and the ASIO thread gets another turn on the next round-robin pass, so a newly-ready fd (an incoming socket event, a fired timer, the eventfd a signal handler writes) is picked up on the following turn. This does mean that a program which sits fully idle waiting on real external I/O (no actor work, a noisy event holding it open) will spin the round-robin at 100% CPU instead of sleeping. (If the awaited I/O never arrives, the run pins a core until a watchdog kills it — the same non-termination the 10ms wait had, only spinning rather than sleeping.) But that case is real-I/O-bound and therefore already outside systematic testing's deterministic envelope, so it is not a scenario we optimize for.

The backends are platform-exclusive — `epoll.c` (Linux), `kqueue.c` (BSD/macOS), `sock_notify.c` (Windows) — and each is only compiled and run on its own platform, so a change to the poll/yield pattern in one is invisible to the others. Nothing in normal CI builds `systematic_testing` at all; the Linux pattern is guarded only by the **weekly** `determinism_smoke.py` (see `systematic-testing-send-ordering.md`), and the Windows (`sock_notify.c`) pattern has **no** CI coverage yet — a regression there surfaces only when someone builds `use=systematic_testing` on a Windows host by hand. The `sock_notify.c` trap is placing the `SYSTEMATIC_TESTING_YIELD()` after the `if(!ok) continue;` that follows `GetQueuedCompletionStatusEx` rather than before it: with a zero timeout a `WAIT_TIMEOUT` takes that `continue` on every empty pass, so a yield below it would never run and the asio thread would spin without handing control back. Run on each platform's host: build `use=systematic_testing` and confirm a trivial actor program reaches the "Systematic testing successfully finished!" banner.

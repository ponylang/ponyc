# The Windows systematic park is non-alertable, and relies on no I/O running under systematic testing

Under `use=systematic_testing` on Windows, a thread waiting its turn parks in `SleepConditionVariableSRW(..., 0)` in `src/libponyrt/sched/systematic_testing.c` (in `ponyint_systematic_testing_wait_start`, `ponyint_systematic_testing_start`, and `ponyint_systematic_testing_yield`). That wait is **not** alertable. The normal Windows scheduler suspend (`ponyint_thread_suspend` / `ponyint_thread_join` in `src/libponyrt/platform/threads.c`) deliberately uses an alertable `WaitForSingleObjectEx(..., true)` so that queued APCs — socket-I/O completion routines and the like — run on the thread; the systematic park drops that.

This is correct only because systematic testing runs no I/O and therefore queues no APC to a scheduler or pinned-actor thread: the ASIO thread is never started under systematic testing, `pony_asio_event_create` aborts, and there is no `QueueUserAPC` onto these threads. If a future change runs overlapped I/O with a completion routine, or queues a user APC, onto a scheduler/pinned thread while systematic testing is active, the non-alertable park will silently swallow it — the APC never runs and the thread waits on with no local signal that anything is wrong. The dependency is enforced entirely in `src/libponyrt/asio/` (I/O is disabled there), not in `systematic_testing.c` where the non-alertable park lives, so the break would not surface near either change.

This is **not** caught per-PR: nothing in normal CI builds `systematic_testing`; the only standing coverage is the daily generative systematic stress jobs (`.github/workflows/stress-test-generative-systematic-*.yml`), and a missed APC surfaces only as a watchdog timeout with no thread state. If you re-enable any I/O path under systematic testing, restore an alertable wait at the three park sites (or otherwise service the APCs) rather than assuming the non-alertable park is safe. Run (Windows): `cmake --preset windows-x86-64 -DPONY_USES=systematic_testing && cmake --build --preset windows-x86-64-debug`, then loop the generative harness under `test/rt-stress/generative` with `--ponymaxthreads` >= 2.

# The libs ExternalProjects must be told the compiler explicitly; they don't inherit it

`lib/CMakeLists.txt` builds LLVM/LLD/clang with `add_subdirectory`, which inherits the parent project's `CMAKE_C_COMPILER`/`CMAKE_CXX_COMPILER`, but builds gbenchmark and googletest with `ExternalProject_Add`, which does NOT. An ExternalProject sub-build inherits only the generator; on a single-config generator (Make/Ninja) it picks its compiler from the `CC`/`CXX` environment and ignores a `CMAKE_C_COMPILER`/`CMAKE_CXX_COMPILER` set on the parent project (for example via a preset's `cacheVariables`). So `lib/CMakeLists.txt` forwards the compiler into both ExternalProjects' `CMAKE_ARGS` (`PONYC_LIBS_COMPILER_ARGS`, non-MSVC only — MSVC's compiler rides in on the generator/toolset, which ExternalProject does carry over).

Drop that forwarding — or select the compiler for the libs build through `CMAKE_C_COMPILER`/`CMAKE_CXX_COMPILER` (a preset cache var, `-D` on the command line) with no matching `CC`/`CXX` in the environment — and gtest/gbenchmark silently build with the environment's default `cc` (gcc on the Linux CI images) while LLVM builds with clang. The mismatch is invisible on x86-64. On aarch64 it breaks the debug link: gcc's `-fpic` `gtest-all.cc.o` references `__stack_chk_guard` through the small-model GOT (`R_AARCH64_LD64_GOTPAGE_LO15`), which overflows once `libponyc.tests` links all of LLVM plus gtest — "too many GOT entries for -fpic". clang's `-fpic` gtest does not. This is why `lib/CMakePresets.json` sets the compiler as a cache var and the libs build still needs the forwarding: the old `make libs` / `build-libs.py` path set `CC`/`CXX` in the environment, so the ExternalProjects inherited it and the coupling was latent.

Run (aarch64, where the break surfaces): the debug build in ponyc-tier2.yml's "arm64 Linux glibc" / "arm64 Linux musl" jobs (`cmake --preset armv8-a-debug && cmake --build --preset armv8-a-debug`) against a freshly built libs cache — i.e. after any change to `lib/CMakeLists.txt`, `lib/CMakePresets.json`, or the libs patches, which re-key the cache and rebuild the libs from source.

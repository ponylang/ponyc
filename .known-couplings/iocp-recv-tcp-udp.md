# Windows IOCP `IOCP_RECV` completion arm is shared by TCP and UDP, and its byte count couples to both `_complete_reads`

The `IOCP_RECV` case in `iocp_callback` (`src/libponyrt/lang/socket.c`) handles completions for both `iocp_recv` (TCP) and `iocp_recvfrom` (UDP), and the count it passes to `pony_asio_event_send(ev, ASIO_READ, count)` is interpreted differently by each consumer's `_complete_reads`: `TCPConnection._complete_reads` (`packages/net/tcp_connection.pony`) treats `count == 0` as a remote close, while `UDPSocket._complete_reads` (`packages/net/udp_socket.pony`) ignores `count` for close detection (it keys on `_read_buf.space() == 0`) and feeds `count` straight into `data.truncate(count)`. The arm special-cases the NTSTATUS `STATUS_BUFFER_OVERFLOW` (oversized UDP datagram) to dispatch the bytes-copied count rather than 0, which relies on TCP stream reads never raising that status (a byte stream has no message boundary to overflow). A change here is invisible to a Linux/macOS maintainer and won't be caught by tests local to either protocol — TCP changes can silently corrupt UDP delivery and vice versa. Run on a Windows host: the `net` tests in `make test` (`stdlib-debug` and `stdlib-release`), which include the UDP buffer-size and TCP receive tests.
